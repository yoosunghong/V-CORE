# Integrated Path/Action SFT Result

Date: 2026-06-15

## Goal

Train and benchmark a single control-plane model that selects both:

- the path/route, such as `process_status`, `simulation_status`, `robot_command`, or `no_action`
- the action/tool for `robot_command`, such as `start_simulation`, `move_to_station`, or `run_station_task`

This experiment intentionally excludes general chat, report generation, and free-form assistant behavior. It should be treated as a compact router/action selector, not a replacement for the general response model.

## llama.cpp Export Note

The initial GGUF conversion produced a model that failed to load in llama.cpp with:

```text
missing tensor 'blk.0.ssm_dt'
```

The active llama.cpp converter in `C:/tmp/llama.cpp/conversion/qwen.py` mapped Qwen3.5 SSM dt tensors to `.dt_proj.bias`; this needed to be mapped to `.dt_proj` for the current llama.cpp loader. After patching the converter and regenerating the GGUF, llama.cpp loaded the Q5_K_M model, generated from `/completion`, and answered a minimal OpenAI-compatible `/v1/chat/completions` request.

The smoke test confirms the quantized file loads. A full OpenAI-compatible chat request with the integrated system prompt returned `413 Payload Too Large` in this local llama-server build, even though the prompt is only 661 characters. Production inference should therefore either use a `/completion` prompt wrapper or adjust the llama-server chat-template/request path, then run the evaluator against the llama.cpp endpoint before replacing the existing router.

## Chat / Report Regression Test (vs standard model)

Date: 2026-06-15

To check whether this router-only SFT can also serve the general chat and report paths
(the tasks `RoutingSplitLlmGateway` deliberately keeps on the general model), the SFT
Q5_K_M GGUF was served on llama.cpp (`:8090`, `-c 8192`) and run through the **real
backend code path** — `LlamaCppLlmGateway.generate_report` / `generate_chat_response`
with the production `report_system`/`report_user` templates and the in-code chat system
prompt — against the standard `qwen3.5:2b` on Ollama. 4 report cases + 4 chat cases each.

Harness: `docs/sft/integrated/scripts/compare_chat_report.py`.
Raw outputs: `docs/sft/integrated/data/compare_chat_report.json`.

| Task | Standard `qwen3.5:2b` | Path/Action SFT |
| --- | --- | --- |
| All 8 calls succeed (no 413) | yes | yes (needs explicit `-c 8192`) |
| Report avg latency | 8.2 s (max 16.5 s) | 2.1 s (max 3.1 s) |
| Chat avg latency | 2.5 s | 1.5 s |
| Output defects | none | 2 (see below) |

The earlier `413 Payload Too Large` was a context-size default, not a real limit — serving
with `-c 8192` cleared it for the full report prompt.

Most SFT outputs were coherent: PASS/FAIL-first report structure, `AI 종합 평가` section,
and on-topic Korean chat with the off-topic guard intact. But two correctness/stability
regressions appeared that the standard model did not have:

1. **Verbatim duplication** — the `station_task_complete` report rendered the entire body
   twice.
2. **Language leakage** — the chat greeting emitted Chinese tokens (`사용자您好!`).
   A third softer slip: the follow-up chat promised to *execute* a re-run
   (`다시 실행해 드리겠습니다`), contradicting the "guide the user to enter an explicit
   command, do not execute" system instruction.

Latency is not a like-for-like model win: SFT is Q5_K_M on GPU llama.cpp while the baseline
is Q8_0 on Ollama, so the gap reflects quant + serving stack as much as the model.

**Verdict:** the path/action SFT is *usable* but **not satisfactory as a drop-in** for chat
and reporting — the duplication and language-leak defects would surface in a demo. This
confirms the result doc's recommended split: keep this GGUF for route/action selection and
keep the general/report/chat model separate (`LLM_PROVIDER=routing_split`), unless a mixed-task
SFT + regression benchmark is added.

## Single-Model Deployment: Adapter Toggle (implemented)

Date: 2026-06-15

The VRAM budget allows only one model, but the system needs routing + chat + reporting.
Solution: serve **one base (`qwen3.5:2b`) + the routing LoRA on one llama.cpp endpoint**,
and toggle the adapter **per request** — scale 1.0 for tool routing (the SFT router),
scale 0.0 for chat/report/plan (the bare base, which has no chat/report defects). This
collapses the two-model `routing_split` into a single in-VRAM base + a ~22 MB adapter.

Enabler confirmed on the project llama.cpp build (9559): per-request `"lora":[{"id":0,
"scale":N}]` in the `/v1/chat/completions` body applies only to that slot, so a chat call
at 0.0 and a routing call at 1.0 do not interfere.

### Artifacts added
- LoRA→GGUF adapter (f16, 21.8 MB): `docs/sft/integrated/data/vcore-path-action-router-adapter-f16.gguf`
  (converted via `C:/tmp/llama.cpp/convert_lora_to_gguf.py <adapter> --base <merged-fp16>`).
- E2E test: `docs/sft/integrated/scripts/test_adapter_toggle.py`
  (results: `docs/sft/integrated/data/adapter_toggle_test.json`).

### Backend changes
- `LlamaCppLlmGateway` gained an `adapter_scale` param → injects the per-request `lora` field.
- New `PathActionRoutingGateway(LlamaCppLlmGateway)`: speaks this model's `{"route","action",
  "arguments"}` protocol under the `path_action_system` prompt (the deployed tool-planning
  gateway expects `{"name","arguments"}`), maps `robot_command` actions to a `ToolCall`, and
  declines every other route + the internal-only `move_to_station`.
- New prompt template `app/prompts/templates/path_action_system.txt`.
- New provider `LLM_PROVIDER=adapter_toggle` in `container.py`: wires
  `RoutingSplitLlmGateway(general=base@0.0, routing=PathActionRoutingGateway@1.0)`, both at
  the same `LLAMA_CPP_BASE_URL`. Scales overridable via `ADAPTER_ROUTING_SCALE` /
  `ADAPTER_GENERAL_SCALE`.

### Startup ordering (important)
The `:8080` llama-server is a **host process, not managed by docker compose**. If the
backend container starts before `:8080` is reachable, the one-time startup preload used to
fail and leave the LLM permanently `failed` ("All connection attempts failed" in the web
chat). Fixed two ways:
- **Self-healing preload** (`app/main.py`): preload now retries every 5 s, so the backend
  stays `loading` and flips to `ready` automatically once the model server is reachable —
  no manual restart. Verified: kill `:8080` → restart backend → status `loading` (not
  `failed`) → bring `:8080` back → recovers to `ready` on its own.
- **Launcher** (`web/start-demo.ps1`): starts the llama-server (if down), waits for
  `:8080` health, then runs docker compose. Use `-Build` to force an image rebuild.
  ```
  powershell -ExecutionPolicy Bypass -File web/start-demo.ps1 -Build
  ```

### Serving command
```
llama-server.exe \
  -m <qwen3.5:2b base blob> \
  --lora docs/sft/integrated/data/vcore-path-action-router-adapter-f16.gguf \
  --lora-init-without-apply \
  --host 0.0.0.0 --port 8080 -ngl 99 -c 8192 --jinja --reasoning off --reasoning-budget 0
```

| Function | Adapter scale | Result |

| Routing (`propose_tool_call`, 166 cases) | 1.0 | **99.4%** (165/166; the one miss is the documented `ag_count` typo) |

Routing through the integrated gateway matches the standalone SFT eval (99.4%), and the
chat/report defects from the all-SFT test are gone because those paths run on the bare base.
32 backend unit tests still pass. This is the recommended single-model configuration.

## Conclusion

The integrated path/action SFT is viable for the control-plane task set. It dramatically outperformed the baseline on the held-out test set while keeping the false positive action rate at zero.

Because general chat and report generation were excluded from both data and evaluation, this result does not prove that the model can replace the general-purpose Ollama model. The safer architecture is:

- use this integrated SFT GGUF for route/action selection
- keep the general/report/chat model separate unless a broader mixed-task SFT and regression benchmark is created
